# Performance Baselines

This directory stores platform-specific performance baseline JSON files used by the
Threshold Engine (`tools/perf/threshold_engine.py`) for regression detection.

Each file follows the naming convention `<platform>.json`, where `<platform>` is the
`uname`-style identifier (e.g., `linux-x86_64`, `darwin-arm64`).

## Bootstrap Flow (First-Time Baseline Generation)

Baselines **must** be generated by actually running benchmarks on the target CI platform.
Cross-platform backfilling from historical data of other platforms is **not permitted**.

### Steps

1. **Run benchmarks on the target platform.**
   A maintainer triggers the baseline generation on the target CI runner (e.g., via
   `workflow_dispatch` on the nightly-perf workflow with `bootstrap_baseline=true`)
   or executes locally on the runner:

   ```bash
   tools/perf/run_perf_baseline.sh --update-baseline
   ```

   The workflow-dispatch bootstrap path uploads `perf-baseline-<platform>.json` as a CI
   artifact. The local command builds the release binary, runs all sample tiers,
   extracts core metric fields from the Measurement Report, and writes the result to
   `perf/baselines/<platform>.json`.

2. **Submit the baseline via PR.**
   Download the workflow artifact or use the locally generated
   `perf/baselines/<platform>.json`, commit it, and open a pull request so the change is
   auditable and reviewable.

3. **Merge the PR.**
   Once merged, subsequent PR Smoke and Nightly Full CI jobs will use the committed
   baseline for threshold comparison.

## Behavior When No Baseline Exists

Before a baseline file is committed for a given platform:

- The Threshold Engine **skips** comparison and exits with code `0`.
- The Verdict Report will contain `"overall_verdict": "skipped"` with an empty
  `comparison.tiers` object.
- PR Smoke and Nightly Full CI jobs will **not** block — they pass without comparison.
- A warning is printed to stderr:
  `No baseline found for platform <platform>, skipping comparison`.

## Baseline File Schema

Each baseline file contains the following top-level fields:

| Field            | Description                                      |
|------------------|--------------------------------------------------|
| `schema_version` | Schema version string (e.g., `"1.0.0"`)         |
| `timestamp`      | ISO 8601 timestamp of when the baseline was captured |
| `git_commit`     | Git commit hash at the time of capture           |
| `platform`       | Platform identifier (e.g., `linux-x86_64`)       |
| `tiers`          | Object keyed by sample tier name, each containing core metrics |

Core metrics per tier: `p50_ms`, `p95_ms`, `p99_ms`, `peak_memory_bytes`, `req_per_s`,
`input_mb_per_s`. Runtime metadata such as `stage_breakdown`, `iterations`, and `warmup`
are intentionally excluded from baselines.

## Important Notes

- **Do not** manually create or pre-fill baseline files with fabricated data.
- **Do not** copy baselines from one platform to another.
- Baselines should only be updated through the documented bootstrap flow above.
- This directory is under version control (not in `.gitignore`).
