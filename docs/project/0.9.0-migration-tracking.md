## B3 — `needs_block_separator` dead field

Source: `components/rust-converter/src/streaming/state_machine.rs:88`.

Current state (verified 2026-06-29):
- `StructuralStateMachine.needs_block_separator` is a `pub` field on a
  **non-`repr(C)`**, Rust-internal struct. It is NOT exposed across the FFI
  boundary (no `#[repr(C)]`, not in any FFI export). The "FFI ABI stability"
  argument does **not** apply here.
- The field is written in `new()` (line 119) and read only in an `ignored`
  doctest (line 785). Production code never reads it.
- `IncrementalEmitter` (`streaming/emitter.rs:147`) has its own independent
  `needs_block_separator` field that IS read/written in production
  (emitter.rs:541,546,550,585,596,752,758). The state-machine copy is the
  redundant one.
- The existing doc comment (state_machine.rs:84-87) already says
  "Retained for internal API compatibility; safe to remove in the next
  breaking-change release" — this is the accurate rationale (not FFI ABI).

**0.9.0 action**: Remove `StructuralStateMachine.needs_block_separator`
(field + the `needs_block_separator: false` init at line 119 + the doctest
assertion at line 785). This is a Rust-internal struct change, not an ABI
break. Acceptance: streaming state machine + emitter tests pass; no
production reader exists so no behavior change.

**0.8.x verdict**: Retain with the *existing* accurate comment (the comment
already says "internal API compatibility", not FFI ABI — no rewrite needed).
Do not delete in 0.8.x to keep the patch line behavior-stable.

## B9 — Ignored doc-tests

Local count (verified 2026-06-29):
`grep -RIn '```ignore\|#\[ignore\]' components/rust-converter/src components/rust-converter/tests`
→ **97 occurrences**: 96 `` ```ignore `` blocks + 1 `#[ignore]`.

Classification (by sample inspection):
- FFI pointer-lifecycle examples requiring `MarkdownOptions`/`MarkdownResult`
  C-ABI struct construction with raw pointers → legitimately ignored (private
  helpers, unsafe C-ABI context not compilable as a doctest against the public
  API).
- `state_machine.rs` examples using `MemoryBudget` struct field literal
  construction (`MemoryBudget { state_stack: 1024 }`) that depends on
  crate-private field visibility → legitimately ignored.
- `options.rs:184` `decode_options` example — private (`pub(crate)`) function,
  not callable from a doctest → legitimately ignored.

**0.8.x verdict**: No change. Removing the `ignore` fences would require
either making private items `pub` (API surface widening) or rewriting examples
to use only the public API. That is a 0.9.0 documentation-quality task, not a
0.8.x release-line cleanup.

**0.9.0 action**: Review each `` ```ignore `` block; convert the ones that can
be expressed purely through the public API to compiled doctests. Keep FFI
pointer-lifecycle / private-helper / unsafe-ABI examples ignored with a
one-line reason comment (`// ignored: requires crate-private ...`).
Acceptance: `cargo test --doc` passes for the project; harness recommendation —
add a `make test-doctest` target that runs `cargo test --doc -p
nginx-markdown-converter` and reports the ignored count, so the count is
tracked across releases rather than rediscovered by external review.

Tracking ID: `DOCTEST-CLEANUP`.
