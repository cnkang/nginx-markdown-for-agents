# Release Notes: 0.9.0

**Date**: 2026-07-06
**Status**: Released
**Type**: Breaking Release (last breaking opportunity before 1.0.0 freeze)

---

## Breaking Changes

### Reason code naming convention changed to lowercase snake_case

All 18 existing reason code strings have been renamed from UPPERCASE_SNAKE_CASE
to lowercase_snake_case. This affects Prometheus label values, structured logs,
and the diagnostics endpoint.

**Examples**:
- `CONVERTED` â†’ `converted`
- `SKIPPED_ACCEPT` â†’ `skipped_accept`
- `PARSE_TIMEOUT` â†’ `timeout`
- `FFI_CALL_ERROR` â†’ `ffi_panic`

**Impact**: Any dashboard, alert rule, or log parser that matches on reason
code strings must be updated. See the
[Migration Guide](../guides/MIGRATION-0.9.md#observability-reason-code-naming-convention-spec-53)
for the complete mapping table and regex-replace patterns.

### Metrics consolidated into 5 unified families

Per-reason counters (e.g., `markdown_skipped_accept_total`,
`markdown_parse_timeouts_total`) are consolidated into 5 unified metric
families with a `reason` label:

| Family | Purpose |
|--------|---------|
| `markdown_conversions_total` | Successful conversions |
| `markdown_skipped_total` | Skipped requests (by reason) |
| `markdown_errors_total` | All errors (by reason) |
| `markdown_failed_open_total` | Fail-open deliveries |
| `markdown_failed_closed_total` | Fail-closed rejections |

**Impact**: Existing PromQL queries, recording rules, and alert expressions
that reference the old per-reason metric names must be rewritten to use
the unified family + `reason` label pattern.

## New Features

### Diagnostics schema v1 (DRAFT)

The `/nginx-markdown/diagnostics` endpoint now returns a versioned JSON schema
(`schema_version: 1`) with 6 structured sections:

- `decision` â€” last decision state, profile, path mode
- `inflight` â€” current/max/high-watermark/overload counts
- `error` â€” error totals, last error reason
- `streaming` â€” streaming eligibility and counters
- `conditional` â€” conditional request evaluation state
- `etag` â€” ETag generation policy and status

This schema is a DRAFT; it will be frozen at 1.0.0. After freeze, only
additive changes (new optional fields) are permitted.

### Label whitelist enforcement

A strict label whitelist limits Prometheus metric labels to 4 allowed keys:

| Label | Purpose |
|-------|---------|
| `reason` | Decision reason code |
| `profile` | Active processing profile |
| `path_mode` | Request processing path |
| `cache_validation` | Cache validation setting |

High-cardinality labels (`url`, `path`, `user_agent`, `request_id`, etc.)
are explicitly blocked to prevent unbounded time series growth.

### 8 new reason codes

| Code | `as_str()` | Description |
|------|-----------|-------------|
| 18 | `conversion_error` | Generic conversion failure |
| 19 | `memory_budget_exceeded` | Memory budget exceeded |
| 20 | `overload` | Inflight guard triggered (overload protection) |
| 21 | `invalid_dynconf` | Invalid dynamic configuration |
| 22 | `degraded_snapshot` | Degraded dynconf snapshot |
| 23 | `header_plan_apply_error` | Header plan commit failure |
| 24 | `streaming_mid_flight_error` | Streaming mid-flight error |
| 25 | `bypass_no_transform` | Cache-Control no-transform bypass (conditional requests) |

These codes provide fine-grained visibility into failure modes that were
previously grouped under coarser categories.
